# Project Analyzer

分析任意软件项目，产出结构化学习文档。目标：让开发者在 30 分钟内理解项目并能讲给别人听。

## 工作流

分析分 6 个阶段，严格按顺序执行。

### 阶段 1: 输入解析

确定项目位置和类型。

**项目位置**：
- 本地路径：直接使用
- GitHub URL 或 `owner/repo`：用 `gh repo clone` 克隆到临时目录

**项目分类**（读取 package.json / Cargo.toml / go.mod / pyproject.toml 检测）：

| 类型 | 检测信号 | 分析侧重点 |
|------|----------|------------|
| CLI 工具 | `bin` 字段、commander/yargs 依赖 | 命令注册、参数解析管道、子命令分发 |
| 脚手架 | `create-*` 命名、模板目录、ejs 依赖 | 模板选择策略、变量注入、生成流程 |
| 组件库 | storybook、src/components、大量 exports | 组件 API、主题系统、组合模式 |
| 框架/库 | lib 输出、types 字段、高抽象层 | 核心抽象、扩展点、生命周期钩子 |
| 全栈应用 | 前后端同存、API 路由、数据库连接 | 请求生命周期、数据层、前后端交互 |
| Monorepo | workspaces、packages/ 目录 | 包间依赖、共享构建、发布策略 |

分类后，读取 `references/strategies.md` 获取对应分析策略。

**确定输出位置**：默认 `{project_dir}/project-analysis/`，用户可指定。

### 阶段 2: 结构扫描 → `01-structure.md`

快速扫描项目骨架。读取 `references/scan-commands.md` 获取扫描命令模板。

**产出内容**：
1. 精简目录树（3-4 层，排除 node_modules/.git/dist）
2. 技术栈速查表（语言、框架、构建工具、测试框架、主要依赖）
3. 入口文件清单（每个入口的职责和调用者）
4. 目录职责映射（每个顶级目录做什么）

**关键动作**：
- 用 Glob 扫描目录结构
- 用 Read 读取清单文件（package.json 等）
- 用 Grep 识别入口文件
- 用 Bash 读取 git log（最近提交、贡献者数量）

### 阶段 3: 架构深潜 → `02-architecture.md`（核心阶段）

这是最重要的阶段。深入代码阅读，理解架构设计。读取 `references/reading-strategies.md`。

**必须产出**：
1. **架构模式描述** — 一句话说清整体模式（分层/插件/管道/MVC/微内核...）
2. **Mermaid 架构图** — 核心模块关系（graph TD）
3. **3-7 个关键模块详解**，每个包含：
   - 职责（一段话说清楚）
   - 核心类型/数据结构（带代码片段）
   - 关键函数签名及其作用（标注文件路径和行号）
   - 与其他模块的交互方式
   - 设计意图（为什么这样设计）
4. **2-3 条核心数据流**（Mermaid sequence diagram 或 flowchart）
5. **设计模式清单**（模式名、使用位置、目的）

**分析策略**按项目类型调整权重，详见 `references/strategies.md`。

**代码阅读顺序**：
- CLI / 脚手架：从入口文件开始，追踪命令分发链（自顶向下）
- 组件库：从核心类型定义开始，理解组件 API 设计（自底向上）
- 框架：先理解生命周期和配置系统，再深入扩展点
- 全栈：从请求生命周期开始（自顶向下）

### 阶段 4: 实践指南 → `03-guide.md`

从"理解"转向"能用"。

**必须产出**：
1. **5 分钟快速上手** — 克隆→安装→运行→看到效果（命令从实际扫描结果生成，非模板占位符）
2. **开发环境搭建** — 完整配置步骤
3. **如何添加一个新功能** — 以具体例子走一遍完整开发流程
4. **如何调试** — 常用调试技巧
5. **如何运行测试** — 测试框架、运行命令
6. **项目约定** — 代码风格、命名规范、PR 流程
7. **常见陷阱** — 新人容易踩的坑、已知问题

### 阶段 5: 知识卡片 → `04-cards.md`

提取可独立消费的知识点，用于"讲给别人听"。

**必须产出**：
1. **电梯演讲**（30 秒说清这个项目）
   - 模板："[项目名] 是一个 [类型]，它通过 [核心机制] 解决了 [问题]。核心创新在于 [关键设计决策]。"
2. **白板讲解**（5 分钟画图讲解架构）— 描述要画什么、怎么讲
3. **技术选型解读** — 为什么选这个技术栈
4. **核心概念速查表** — 术语→一句话解释→对应代码位置
5. **对比学习** — "如果你熟悉 X，那么这个项目中的 Y 相当于..."

### 阶段 6: 质量审查

读取 `references/quality-checklist.md` 执行完整质检。

**核心检查**：
- 架构图中的模块关系有代码证据支撑
- 所有文件路径真实存在（不是猜测）
- 快速上手命令可直接复制执行
- 每个"为什么"都有证据支撑
- 无推测性内容（不确定的标注为"推测"）

质检通过后，写 `README.md` 作为索引：
- 项目一句话定义 + 电梯演讲
- 技术栈速查表
- 各章节链接和预估阅读时间
- 分析信息（日期、项目版本/commit、项目类型）

## 输出结构

```
project-analysis/
├── README.md              # 索引 + 技术栈速查 + 章节导航
├── 01-structure.md        # 项目结构、技术栈、入口文件、目录职责
├── 02-architecture.md     # 架构模式、模块详解、数据流、设计模式
├── 03-guide.md            # 快速上手、开发流程、陷阱
└── 04-cards.md            # 电梯演讲、白板讲解、概念速查
```

## 写作风格

- 面向有经验的开发者，不解释基础概念
- 代码路径用反引号标注，如 `src/core/plugin.ts`
- 用具体代码片段代替抽象描述
- 每个"为什么"都要有证据（代码或文档证据）
- 不说废话：删掉不影响理解的段落
- 中文写作，技术术语保留英文
- 不出现 AI 味套话（"值得注意的是"、"综上所述"、"总而言之"）
