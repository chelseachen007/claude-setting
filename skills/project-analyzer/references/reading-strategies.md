# 代码阅读策略

如何高效地理解一个陌生项目的代码。

---

## 阅读顺序

### 自顶向下法（推荐用于 CLI / 脚手架 / 全栈）

适用场景：项目有明确的入口点和线性执行流。

1. 从入口文件开始（`src/index.ts`、`src/cli.ts`、`main.go` 等）
2. 追踪主要的调用链（入口函数调用了什么）
3. 在分支点选择最重要的路径深入
4. 重复直到覆盖核心模块
5. 最后回过头处理工具函数和辅助模块

**技巧**：用 Grep 搜索入口函数名，找到所有调用链的起点。

### 自底向上法（推荐用于组件库 / 库）

适用场景：项目的价值在于其抽象和 API 设计。

1. 从核心类型定义开始（`types.ts`、`interfaces.ts`、`src/types/`）
2. 理解核心数据模型和抽象
3. 追踪这些类型如何被具体实现消费
4. 理解对外暴露的 API 层
5. 最后看构建和打包逻辑

**技巧**：从 `index.ts` 的导出开始，这些是项目认为最重要的东西。

### 混合法（推荐用于框架 / 大型项目）

适用场景：项目复杂度高，单一方向不够。

1. 先自顶向下，建立全局视图（入口 → 主要模块）
2. 在关键模块处切换为自底向上（深入理解模块内部）
3. 确保理解每个模块的"接口契约"（输入什么、输出什么）
4. 用数据流串联各模块的理解

## 识别核心模块的信号

以下信号表明一个文件/目录是核心模块：

1. **被最多模块 import** — 用 Grep 搜索 `from './module'` 或 `import './module'` 的出现频率
2. **包含核心数据结构/类型** — 文件名含 `types`、`model`、`schema`、`interface`
3. **导出最多符号** — `export` 语句数量多
4. **命名暗示** — 文件名含 `core`、`engine`、`kernel`、`main`、`index`、`app`
5. **README 单独提到** — 文档中专门有章节讨论的模块
6. **最深的调用栈** — 处于调用链中间层的模块通常最核心

### 排除噪音的信号

以下文件通常不是核心模块：
- `utils/`、`helpers/`、`constants/` — 工具函数，理解即可，不需深潜
- `__tests__/`、`*.test.*`、`*.spec.*` — 测试文件，用于验证理解
- `types/`（仅导出类型） — 类型定义，需要读但不需要追踪执行流
- 配置文件 — 了解配置项即可

## 追踪调用链的技巧

### 正向追踪（从调用者到被调用者）

1. 从入口函数开始
2. 用 Grep 搜索函数名找到所有定义和调用
3. 对每个调用点，读取上下文理解调用时机
4. 递归追踪到叶子函数

### 反向追踪（从被调用者到调用者）

1. 从一个关键函数/类开始
2. 用 Grep 搜索谁调用了它（搜索函数名/类名）
3. 理解每个调用者的场景
4. 向上追踪到入口

### 依赖追踪

1. 搜索 import/require 语句理解模块依赖
2. 画出依赖方向图（A imports B → A 依赖 B）
3. 寻找循环依赖（通常意味着需要重构或有特殊设计）

## 理解设计意图的信号

当问"为什么这样设计"时，从这些信号寻找答案：

### 代码内的信号

- **注释** — 尤其是解释 "why" 的注释（"We do X because..."）
- **提交信息** — `git log --follow -p -- src/core/engine.ts` 看文件的演变历史
- **测试用例** — 测试展示了预期用法和边界情况
- **错误处理** — 错误消息和错误类型揭示了设计者预期的使用场景
- **命名** — 函数名和变量名反映了设计者的思维模型

### 代码外的信号

- **README** — "设计哲学"、"架构决策" 部分
- **Issue / PR 讨论** — 设计权衡的原始记录
- **CHANGELOG** — 重大变更的方向
- **CONTRIBUTING.md** — 项目约定的来源

### 推断设计意图的模式

- **适配器模式** 出现 → 说明需要支持多种后端/实现
- **中间件/管道** 出现 → 说明需要可组合的处理链
- **插件系统** 出现 → 说明核心功能需要可扩展
- **配置驱动** 出现 → 说明行为需要高度可定制
- **事件/观察者** 出现 → 说明模块间需要松耦合通信

## 大型项目的分析节奏

对于超过 100 个源文件的项目：

1. **第一轮（5 分钟）**：只看目录结构 + README + package.json，建立高层理解
2. **第二轮（10 分钟）**：读入口文件 + 核心模块的 index.ts，理解模块边界
3. **第三轮（15 分钟）**：深入 3-5 个核心模块的内部实现
4. **第四轮（5 分钟）**：验证理解，补充遗漏

每轮结束后问自己：能用一句话说清这个模块做什么吗？如果不能，还需要继续读。
