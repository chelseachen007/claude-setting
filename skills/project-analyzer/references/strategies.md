# 项目类型分析策略

每种项目类型的分析重点和核心数据流模板。

---

## CLI 工具

**检测信号**: `bin` 字段、commander/yargs/ink 依赖、`#!/usr/bin/env` shebang

**分析重点**:
- 命令注册机制（如何定义子命令）
- 参数解析流程（命令行参数 → 内部数据结构）
- 输出格式化（终端彩色输出、表格、进度条）
- 配置加载链（命令行参数 > 环境变量 > 配置文件 > 默认值）
- 插件/扩展机制（如果存在）

**核心数据流**: 用户输入 → 参数解析 → 命令匹配 → 执行 → 输出格式化 → 退出

**关键文件模式**: `src/cli.ts`, `src/commands/**/*.ts`, `src/utils/*.ts`, `bin/*`

**关键模块识别**:
- 命令分发器（路由命令到处理函数）
- 参数解析器（定义和验证参数）
- 渲染器（格式化输出到终端）
- 配置管理器（加载和合并配置源）

---

## 脚手架

**检测信号**: `create-*` 命名、`templates/` 目录、ejs/handlebars/pug 依赖、`inquirer`/`prompts` 交互

**分析重点**:
- 模板选择策略（交互式问答 → 模板匹配）
- 模板引擎和变量注入机制
- 依赖安装策略
- 后处理钩子（安装后自动执行的操作）
- 预设/模板扩展机制

**核心数据流**: 用户交互 → 模板选择 → 变量收集 → 文件生成 → 依赖安装 → 后处理

**关键文件模式**: `src/generator.*`, `src/prompts.*`, `templates/**`, `src/plugins/*`

**关键模块识别**:
- 问答引擎（收集用户偏好）
- 模板引擎（渲染模板文件）
- 文件系统操作（创建目录和文件）
- 包管理器适配（npm/yarn/pnpm）

---

## 组件库

**检测信号**: storybook 配置、`src/components/`、`exports` 含大量组件、CSS-in-JS/styled/Tailwind 插件

**分析重点**:
- 组件 API 设计模式（props/slots/events）
- 主题系统实现（CSS 变量 / CSS-in-JS / Design Tokens）
- 组件注册和按需加载机制
- 类型导出策略
- 文档系统（Storybook / MDX）
- 组件组合模式

**核心数据流**: 组件定义 → 类型推导 → 样式注入 → 渲染 → 事件冒泡

**关键文件模式**: `src/components/**/*.vue`, `src/theme/**`, `src/index.ts`, `.storybook/**`

**关键模块识别**:
- 组件基础层（共享逻辑、mixin、composable）
- 主题引擎（token 系统、样式变量）
- 类型系统（泛型推导、组件类型）
- 构建管道（打包、tree-shaking）

---

## 框架/库

**检测信号**: `lib` 输出、`types` 字段、高抽象层、`plugin`/`middleware`/`hook` 关键词

**分析重点**:
- 核心抽象层（哪些概念是用户必须理解的）
- 生命周期钩子（初始化 → 运行 → 销毁）
- 插件/中间件系统
- 配置系统（用户如何配置行为）
- 扩展点（用户如何扩展功能）

**核心数据流**: 配置加载 → 初始化 → 插件注册 → 请求/事件循环 → 清理

**关键文件模式**: `src/core/**`, `src/plugins/**`, `src/config.*`, `src/types.*`

**关键模块识别**:
- 核心引擎（主循环、生命周期管理）
- 插件系统（注册、加载、执行插件）
- 配置系统（加载、验证、合并配置）
- API 层（对外暴露的公共接口）

---

## 全栈应用

**检测信号**: 前后端同存、API 路由、数据库连接、认证中间件

**分析重点**:
- 请求生命周期（从前端到数据库再回来）
- 前后端数据契约（API 层、类型共享）
- 状态管理策略
- 认证/授权流程
- 数据库模式和迁移

**核心数据流**: HTTP 请求 → 中间件链 → 路由匹配 → 业务逻辑 → 数据层 → 响应序列化

**关键文件模式**: `src/routes/**`, `src/middleware/**`, `src/models/**`, `src/controllers/**`

**关键模块识别**:
- 路由层（URL 映射到处理函数）
- 中间件链（认证、日志、错误处理）
- 数据层（ORM、查询构建器、缓存）
- API 契约（请求/响应类型定义）

---

## Monorepo

**检测信号**: `workspaces` 配置、`packages/` 目录、lerna/turborepo/nx 依赖

**分析重点**:
- 包间依赖关系图
- 共享构建配置
- 发布策略（独立版本 / 固定版本）
- 开发工作流（如何同时开发多个包）

**核心数据流**: 源码变更 → 增量构建 → 影响范围分析 → 测试 → 发布

**关键文件模式**: `packages/*/package.json`, `scripts/**`, 配置文件根目录

**关键模块识别**:
- 构建编排（构建顺序、增量构建）
- 包间通信（共享类型、工具函数）
- 发布管理（版本管理、changelog）
- 开发工具（热更新、调试配置）

**分析策略**: 对 Monorepo 只做包级别概览 + 重点包（2-3 个核心包）深潜，不全量分析。

---

## 类型判定优先级

当项目符合多个类型时，按以下优先级判定：
1. Monorepo（有 workspaces 就算）
2. 全栈应用（有前后端 + API 路由）
3. 框架/库（有 plugin/hook/middleware 系统）
4. CLI 工具（有 bin 字段）
5. 脚手架（有 create-* 命名或模板目录）
6. 组件库（有 storybook 或组件目录）

优先级高的类型决定分析策略，但可以同时提及其他类型的特征。
