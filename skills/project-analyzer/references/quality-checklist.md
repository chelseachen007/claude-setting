# 质量审查清单

在阶段 6 执行的完整质检。逐项检查，不通过的必须修正后才能交付。

---

## 完整性检查

- [ ] 项目类型判定准确（检测信号与判定结果一致）
- [ ] 技术栈清单完整（语言、框架、构建工具、测试框架、主要依赖）
- [ ] 目录结构图反映实际（无遗漏重要目录，无虚构目录）
- [ ] 关键模块数量合理（3-7 个，覆盖核心功能）
- [ ] 数据流覆盖了最重要的使用场景（至少 2 条）
- [ ] 快速上手指南包含完整的命令序列

## 准确性检查

- [ ] 架构模式描述与代码实现一致（不是套用模式名称）
- [ ] Mermaid 图中的关系有代码证据支撑（每个箭头都能追踪到具体代码）
- [ ] 所有文件路径真实存在（用 Glob 验证，不是猜测的路径）
- [ ] 设计模式识别准确（有具体代码位置，不是强行套用）
- [ ] 关键函数签名与实际代码一致（参数、返回值）
- [ ] 依赖关系描述正确（A 依赖 B 在代码中确实存在）

## 可操作性检查

- [ ] 快速上手的命令可直接复制执行（从实际扫描结果生成，非模板）
- [ ] 每个关键模块都有代码片段示例（不是纯文字描述）
- [ ] "如何添加新功能"有完整步骤（不是泛泛而谈）
- [ ] 常见陷阱有具体的错误信息或症状（不是"注意 XX"）
- [ ] 测试运行命令经过验证（与 package.json scripts 一致）

## 可讲解性检查

- [ ] 电梯演讲能让不了解此项目的人在 30 秒内理解项目价值
- [ ] 白板讲解能在 5 分钟内画出核心架构（有明确的步骤）
- [ ] 概念速查表覆盖了所有核心术语（无遗漏重要概念）
- [ ] 有"对比学习"（类比已知项目帮助理解）
- [ ] 技术选型解读有理由（不只是"用了 XX"）

## 写作质量检查

- [ ] 无 AI 味套话（搜索并删除："值得注意的是"、"综上所述"、"总而言之"、"不言而喻"、"众所周知"）
- [ ] 每段信息都有增量价值（删除重复内容）
- [ ] 代码路径用反引号标注（如 `src/core/plugin.ts`）
- [ ] 中文技术文档使用常见英文术语（不生硬翻译，如"中间件"不翻译为"middleware"）
- [ ] 代码片段有注释说明关键部分（不是大段无解释的代码）
- [ ] Mermaid 图能正确渲染（语法正确，无遗漏分号）

## 推测标注检查

- [ ] 所有基于代码证据的断言，没有特别标注
- [ ] 所有推测性的断言，明确标注"[推测]"（如"这个设计可能是为了...[推测]"）
- [ ] 推测性内容不超过总内容的 10%
- [ ] 关键结论不依赖于推测性内容

## 安全检查

- [ ] 文档中不包含 .env 内容、密钥、token
- [ ] 不包含内部 URL（如 http://172.16.x.x）
- [ ] 不包含个人邮箱或手机号（除非是公开的 maintainer 信息）
